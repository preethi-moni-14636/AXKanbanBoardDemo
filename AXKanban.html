A Playbook for Building with AI

Built with AI agents.

One prompt grew into a coordinated workflow — four AI agents, six repeatable steps, and a human at every merge. Here is the playbook: how we contracted the work, where AI led, and where humans stayed accountable.

Claude Copilot 4 agents 6 steps

Working Principles

Spec first

Every feature was written down before code. The spec named every component and design token — no subjective calls at review.

Plan before code

Each feature started with a planning pass: the agent traced files and produced a checklist. Code only started after approval.

Humans own the merge

Agents flagged spec gaps and off-spec choices. Every change was reviewed and pushed by a human. The pipeline ends with a human.

Repository Architecture

The AI spec and agent configuration sit in the parent repo. Each developer's platform project lives inside that folder as a separate git repo, kept out of the parent's index by .gitignore.

Constraints

Shared component library

The shared toolkit — AXText, AXButton, AXIconButton, AXMessage, AXBanner, AXToast, AXLinearProgressBar, AXTextChip — and the *Data / *Properties split on every component. Set once by the team; agents referenced it thousands of times.

Design tokens

Every color and font comes from a Focas sys_* token. Hardcoded hex values or platform-default fonts were a spec violation — agents were instructed to flag them on sight. Zero escaped to main.

State architecture

The KBIntent / KBViewState / KBEvent MVI state model was defined by engineers before Phase 1 began. Agents could execute within it but never deviate from it.

Human-owned merge

No AI ever had commit rights. Every change — including AI-generated test files — required a human to stage, review, and push. The pipeline always ends with a human decision.

Every decision above was set once by humans, then referenced thousands of times by agents. Agents executed inside those boundaries — never across them.
